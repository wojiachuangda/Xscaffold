# Agentic App Platform 技术开发文档

## 1. 项目概述

- **项目名称**：Agentic App Platform
- **目标**：构建支持智能化、模块化应用开发的平台，用户通过声明式配置（YAML/JSON）和可组合智能组件快速搭建、部署和运行具备自主决策能力的应用。
- **使用场景**：开发人员/业务专家定义智能应用的工作流、数据源和决策逻辑；平台自动编排Agent、执行任务、管理上下文并集成外部系统。
- **用户群体**：全栈开发工程师、AI应用开发者、业务架构师、数据科学家。

## 2. 技术栈

| 类别 | 技术选型 | 用途 |
|------|----------|------|
| 运行时 | Node.js (v20+) | 核心服务 |
| HTTP框架 | Express.js / Fastify | API网关、Webhook |
| 数据存储 | SQLite / PostgreSQL | Agent元数据、配置、执行记录 |
| 缓存/队列 | Redis + BullMQ | 会话记忆、分布式锁、异步任务 |
| 可观测性 | OpenTelemetry | 追踪、指标 |
| 校验 | Zod | 配置、参数校验 |
| 编排辅助 | LangChain.js (可选) | Agent编排增强 |
| 前端 | Vue.js | 管理界面 |

## 4. 功能域详细设计

### 4.1 智能体管理 (高优先级)

| 属性 | 内容 |
|------|------|
| **模块名** | `agentManager` |
| **依赖** | 无 |
| **职责** | Agent的CRUD、元数据存储、状态管理、运行时获取接口 |
| **不负责** | Agent执行逻辑、消息历史、工作流编排、前端UI |

#### 核心数据结构 (agentModel)

```typescript
interface Agent {
  id: string;
  name: string;
  description?: string;
  model: string;              // 模型标识，如 "gpt-3.5-turbo"
  tools: string[];            // 工具名称列表
  status: 'enabled' | 'disabled';
  createdAt: Date;
  updatedAt: Date;
}
```

#### 核心服务 (agentService)

| 函数名 | 说明 |
|--------|------|
| `createAgent(data)` | 创建Agent |
| `updateAgent(id, data)` | 更新Agent |
| `deleteAgent(id)` | 删除Agent |
| `getAgentById(id)` | 获取单个Agent |
| `listAgents(filter)` | 列出Agent（支持分页、状态过滤） |

#### 验收标准

- [x] 提供完整REST API（POST /agents, GET /agents/:id等）
- [x] 服务重启后Agent数据正常加载
- [x] 删除后无法再通过接口获取

### 4.2 工作流编排引擎 (高优先级)

| 属性 | 内容 |
|------|------|
| **模块名** | `workflowEngine` |
| **依赖** | `agentManager` |
| **职责** | 解析DAG定义、拓扑执行、数据传递、条件/循环/并行、重试与超时 |
| **不负责** | Agent内部推理、配置持久化、可视化编辑 |

#### 核心组件

**workflowExecutor** - 主执行器

| 函数名 | 说明 |
|--------|------|
| `execute(workflowDef, context)` | 执行整个工作流 |
| `executeNode(node, input)` | 执行单个节点 |
| `handleCondition(expr, context)` | 条件分支判断 |
| `retryNode(node, error, retryCount)` | 节点重试逻辑 |

**nodeRunner** - 节点执行器

| 函数名 | 说明 |
|--------|------|
| `runAgentNode(node, input)` | 调用Agent执行推理 |
| `runToolNode(node, input)` | 执行工具函数 |
| `runCodeNode(node, input)` | 执行自定义代码片段 |

#### 工作流定义示例

```json
{
  "nodes": [
    { "id": "start", "type": "agent", "agentId": "planner" },
    { "id": "decision", "type": "condition", "expression": "{{start.output.score > 0.5}}" },
    { "id": "high", "type": "tool", "toolName": "sendEmail" },
    { "id": "low", "type": "agent", "agentId": "escalate" }
  ],
  "edges": [
    { "from": "start", "to": "decision" },
    { "from": "decision", "to": "high", "condition": "true" },
    { "from": "decision", "to": "low", "condition": "false" }
  ]
}
```

#### 验收标准

- [x] 顺序执行两个Agent节点，第二个能读取第一个输出
- [x] 条件分支根据表达式选择正确路径
- [x] 节点失败按配置重试3次

### 4.3 应用配置管理 (高优先级)

| 属性 | 内容 |
|------|------|
| **模块名** | `configManager` |
| **依赖** | 无 |
| **职责** | 解析/校验YAML/JSON、转换为工作流定义、存储与热加载 |
| **不负责** | 执行工作流、验证Agent存在性 |

#### 核心组件 (configLoader)

| 函数名 | 说明 |
|--------|------|
| `loadFromFile(filePath)` | 从文件加载配置 |
| `validateSchema(config)` | 校验配置结构（Zod） |
| `toWorkflowDef(config)` | 转换为workflowEngine接受的格式 |

#### YAML配置示例

```yaml
name: customerSupport
version: 1.0
nodes:
  - id: classify
    type: agent
    agentId: intent-classifier
  - id: route
    type: condition
    expression: "{{classify.output.intent}} == 'refund'"
  - id: refund
    type: workflow
    ref: refund-process
```

#### 验收标准

- [x] 正确解析合法YAML并生成workflowDef
- [x] 缺失必要字段抛出自定义异常

### 4.4 工具库与插件系统 (高优先级)

| 属性 | 内容 |
|------|------|
| **模块名** | `toolRegistry` |
| **依赖** | 无 |
| **职责** | 注册内置工具、加载外部插件、提供工具列表、执行与参数校验 |
| **不负责** | 工作流编排、插件UI、结果持久化 |

#### 核心组件 (toolRegistry)

| 函数名 | 说明 |
|--------|------|
| `register(toolDef)` | 注册工具 |
| `getTool(name)` | 获取工具定义 |
| `listTools()` | 列出所有工具 |
| `executeTool(name, params, context)` | 执行工具并返回结果 |

#### 内置工具 (builtinTools)

| 工具名 | 函数签名 | 说明 |
|--------|----------|------|
| httpRequest | `(url, method, headers, body)` | HTTP请求 |
| queryDatabase | `(connectionId, sql, params)` | 数据库查询 |
| readFile | `(path, encoding)` | 读取文件 |
| addNumbers | `(a, b)` | 两数求和 |
| sendEmail | `(to, subject, body)` | 发送邮件 |

#### 插件加载机制

- 插件目录：`./plugins/`
- 插件格式：导出 `register(toolRegistry)` 函数的npm包或动态代码
- 平台启动时自动扫描并加载

#### 验收标准

- [x] 内置工具自动注册并可调用
- [x] 插件目录下的模块被动态加载

### 4.5 记忆与上下文管理 (中优先级)

| 属性 | 内容 |
|------|------|
| **模块名** | `memoryManager` |
| **依赖** | `agentManager` |
| **职责** | 存储对话历史、上下文摘要、短期/长期记忆、注入记忆给Agent |
| **不负责** | 工作流执行、前端展示 |

#### 核心组件

**memoryStore** - 存储层

| 函数名 | 说明 |
|--------|------|
| `saveMessage(sessionId, role, content)` | 保存消息 |
| `getHistory(sessionId, limit)` | 获取最近N条记录 |
| `clearSession(sessionId)` | 清除会话 |

**contextSummarizer** (可选)

| 函数名 | 说明 |
|--------|------|
| `summarizeIfNeeded(sessionId, threshold)` | 消息超过阈值时调用LLM生成摘要 |

#### 记忆注入机制

Agent执行时，自动从`memoryManager`获取最近`MEMORY_WINDOW_SIZE`条消息，拼接到Prompt中。

#### 验收标准

- [x] 多轮对话中Agent能访问最近3轮历史
- [x] 记忆窗口大小可配置

### 4.6 API网关与Webhook (中优先级)

| 属性 | 内容 |
|------|------|
| **模块名** | `apiGateway` |
| **依赖** | `workflowEngine`, `configManager` |
| **职责** | REST API触发工作流、接收Webhook、认证限流、参数校验 |
| **不负责** | 工作流内部细节、前端管理界面 |

#### API端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/workflows/:id/execute` | 触发工作流，返回executionId |
| GET | `/workflows/executions/:id` | 查询执行状态与结果 |
| POST | `/webhooks/:provider` | 接收外部Webhook（如GitHub） |

#### 限流策略

- 默认：每用户每分钟60次
- 可基于`RATE_LIMIT_PER_MINUTE`配置

#### Webhook处理 (webhookHandler)

| 函数名 | 说明 |
|--------|------|
| `handleIncoming(payload, headers)` | 解析并转换Webhook为标准事件 |
| `verifySignature(payload, signature, secret)` | 验证签名（支持GitHub格式） |

#### 验收标准

- [x] 外部系统可通过API触发工作流并获得同步/异步响应
- [x] 限流配置生效

### 4.7 可观测性与日志 (低优先级)

| 属性 | 内容 |
|------|------|
| **模块名** | `observability` |
| **依赖** | `workflowEngine`, `agentManager` |
| **职责** | 收集trace、记录日志、暴露指标接口、日志查询API |
| **不负责** | 日志持久化策略、用户认证 |

#### 核心组件

**traceCollector** - 追踪收集器

| 函数名 | 说明 |
|--------|------|
| `startTrace(workflowId, executionId)` | 开始追踪 |
| `endTrace(traceId)` | 结束追踪 |
| `addSpan(traceId, name, duration, metadata)` | 添加Span |

**metricsExporter** - 指标导出

| 函数名 | 说明 |
|--------|------|
| `recordExecution(workflowId, duration, status)` | 记录执行 |
| `getStats(timeRange)` | 返回执行次数、耗时、错误率 |

#### 日志级别

- debug: 工作流节点输出快照
- info: 工作流启动/完成、Agent节点执行、工具调用、Webhook接收
- warn: 内存超80%
- error: Agent推理失败、工具超时、配置校验失败

#### 验收标准

- [x] 可获取最近10次工作流执行的详细trace
- [x] 日志级别可配置

|------|----------|----------|

| warn | 内存使用超过80% - {memoryUsage} | 内存告警 |
| info | Webhook接收 - {source, payloadSignature} | 收到回调 |
| error | 配置校验失败 - {configPath, validationErrors} | 加载配置 |
| debug | 工作流节点输出 - {nodeId, outputSnapshot} | 节点完成 |
| info | 工作流完成 - {workflowId, executionId, totalDurationMs} | 执行结束 |

## 6. 非功能性要求

- **可靠性**：节点失败自动重试3次，超时默认30秒可配置
- **可扩展**：工具通过插件机制扩展，无需修改核心代码
- **多租户**：后期支持租户隔离（数据表添加tenant_id）
- **安全性**：API需JWT认证，Webhook支持签名验证
- **可观测**：OpenTelemetry集成，支持导出到Jaeger/Prometheus
