# ⚖️ Xscaffold 最高开发范式红线（RULES）

你必须无条件严格遵循以下三阶段研发流。违反任何一条，你的本次任务都将被直接判定为不合格。

## 阶段 1：PLAN（计划阶段）—— 严禁直接修改代码
* 在你读取完文件后，你必须首先在终端输出或专属文件中输出一份 `PLAN.md`。
* 计划中必须包含：1. 当前现状分析；2. 修改范围评估；3. 预估可能破坏的已有业务。
* 必须在得到用户口头或指令“CONFIRM”后，方可进入下一阶段。

## 阶段 2：SPEC（契约阶段）—— 接口与契约先发
* 在开始写核心 Service 业务之前，你必须使用 Zod 库定义全量数据契约。
* 契约必须严格按照 AA-SEAC V2 规范，存放在 `src/domain/{domain_name}/schemas/` 目录下。

## 阶段 3：CODE（编码阶段）—— 落地实现
* 编写具体的函数与类，且单个函数长度严格控制在 50 行以内。
* 编写完成后，你必须在文件第一行自动补充标准头注释，说清楚它的 Description 职责。

## 改动规模判定

### 小改动
满足全部条件才算小改动：
1. 修改文件不超过 2 个；
2. 不新增 API、不改变已有 API 请求/响应契约；
3. 不新增数据库表、字段、迁移；
4. 不改变工作流执行语义、状态机、权限、安全策略；
5. 不引入新依赖；
6. 不新增目录级架构；
7. 不影响生产配置、启动流程、部署方式；
8. 可通过现有测试覆盖验证。

小改动可以先给简短计划，用户无异议即可执行。

### 大改动
命中任一条件即算大改动：
1. 修改 3 个及以上文件；
2. 新增或修改 REST API / Tool / Workflow 契约；
3. 新增 Zod Schema 或改变现有 Schema；
4. 涉及数据库 migration、Repository、队列、权限、认证、限流；
5. 涉及 UI 独立项目、构建链、路由、状态管理；
6. 引入/升级依赖；
7. 改变目录结构或模块边界；
8. 影响 IOOR、trace、metrics、日志脱敏、安全审计；
9. 需要新增集成测试或 E2E 测试；
10. 任何可能改变现有业务行为的修改。

大改动必须先输出 PLAN，等待用户明确 `CONFIRM` 后才能进入编码。
