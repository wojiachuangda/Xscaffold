# 📖 AI 智能体系统工程架构与行为合规规范 (AA-SEAC)
> **AI Agent System Engineering Architecture & Behavioral Compliance Specifications**
>
> **文件版本**：AA-SEAC-2026-V2 (终极定稿版)  
> **生效日期**：2026-05-18  
> **核心原则**：安全第一、凡动必留痕、最终人负责。

---

## 🛠️ 第一部分：通用基础编码约束 (General Code Standards)

### 1. 命名规范 (Naming Conventions)
* **唯一含义**：禁止使用无意义的单字母命名（仅循环变量 `i`, `j` 允许在局部上下文内例外）。
* **格式统一**：类名必须使用大驼峰（`CamelCase`）。变量与函数名在**单一项目内必须统一**使用小驼峰（`camelCase`）或蛇形（`snake_case`），严禁在同仓储内混用。常数必须全大写并用下划线分割。
* **动宾结构**：函数命名必须清晰体现动作与业务对象，如 `getUserId()`、`calculateTotal()`。
* **禁止拼音**：坚持全英文命名，严禁拼音与英文混用（如禁止使用 `getGongzi()`）。

### 2. 代码格式与风格 (Formatting & Style)
* **缩进统一**：在一个项目内**必须固定**使用 4 个空格或 1 个 Tab，严禁在同仓储内混用。
* **单行长度**：代码单行宽度严格限制在 120 个字符以内，超出部分必须手动或自动换行。
* **大括号换行**：左大括号 `{` 跟随语句不换行，右大括号 `}` 必须独占一行。
* **自动化校验**：项目必须配置代码 Lint 工具（如 ESLint、Pylint、Checkstyle），**未通过本地校验的代码严禁提交与合并**。

### 3. 函数与类设计 (Design Principles)
* **单一职责**：一个函数或类应当只专注于做好一件事情，避免臃肿。
* **行数限制**：单个函数代码严格控制在 50 行以内；单个独立代码文件不超过 500 行。
* **文件头注释**：所有**新建的源代码文件顶部**必须强制包含规范注释，格式为：`// [{角色}] ID: {任务编号} | Date: {YYYY-MM-DD} | Description: {一句话描述该文件核心职责}`。
* **参数限制**：函数输入参数原则上不超过 4 个，超出时必须封装为对象或结构体（DTO/Payload）。
* **控制嵌套**：`if/else` 或循环嵌套深度**严禁超过 3 层**，必须使用卫语句（Guard Clauses）提前返回以降低阅读成本。

### 4. 异常与错误处理 (Error Handling)
* **严禁吞掉异常**：捕获错误（`catch`）后必须进行实质性处理或打印错误日志，**严禁编写空的 catch 块**。
* **精准捕获**：严禁直接捕获基类 `Exception`，必须明确捕获具体的异常类型（如 `NullPointerException`, `IOException`）。
* **空指针防御**：所有外部输入数据、API 返回值、数据库查询结果，在使用其属性或方法前**必须进行非空校验**。
* **资源释放**：流、数据库连接、文件句柄等系统资源，必须在 `finally` 块或 `using/with` 自动关闭语句中显式释放。

### 5. 注释规范 (Comments)
* **解释原因而非过程**：注释应阐述“为什么要这样写（Why）”，而不是机械复述“这段代码在做什么（What）”。
* **公共接口必填**：所有对外的 API 路由、公共方法、核心业务类必须包含标准的文档注释（如 Javadoc、JsDoc）。
* **标记清理**：临时调试代码不得随版本发布。发版前必须检索并清除所有临时标记。无法清除的必须在代码中登记负责人、原因与预计清理版本，格式为：`// TODO: [负责人] [原因] [预计清理版本]`。

### 6. 安全与性能 (Security & Performance)
* **严禁硬编码**：密码、密钥、API Token、数据库连接串及环境配置严禁写入代码，必须统一抽离至配置文件或环境变量。
* **日志脱敏**：严禁在系统日志中打印明文的用户密码、身份证、银行卡等核心敏感隐私信息。
* **循环内避险**：**严禁在循环体内部**进行数据库查询（SQL）、外部网络请求（API）或频繁创建生命周期仅在循环内的临时大对象。

## 第三部分：四大研发硬性约束

### 约束 1：API 统一响应契约（Response Contract）
* **全站强制执行统一响应格式**。所有路由的最终返回结果必须经过标准格式化函数处理。Controller 内部严禁手动拼装错误响应，遇到异常必须统一抛出继承自 `AppError` 的自定义异常，由接口最外层 Express 全局错误中间件兜底拦截。

### 约束 2：强类型请求入参校验（Input Validation）
* **必须引入强类型校验库（如 Zod）**。在流量通过 Route 进入 Controller 之前，必须在中间件中进行强制 Schema 校验拦截。凡不符合输入定义的请求直接拒绝，Service 业务逻辑内部永远只处理合法数据。

### 约束 3：独立状态机隔离（State Machine Isolation）
* **必须将状态流转逻辑完全剥离为独立的状态机文件（如 taskStateMachine.js）**，对外仅暴露 `transition(currentState, action)` 纯函数。Repository 层的更新方法必须对入参状态进行合法性断言，严禁直接在 SQL 语句中硬编码或绕过状态机修改状态字段。

### 约束 4：依赖倒置与 Repository 模式 (Dependency Inversion)
* **所有的底层存储层操作必须封装在特定的 Repository 类中**，上层业务层只允许调用高度抽象的语义化接口（如 `taskRepository.findById()`）。业务层严禁感知底层存储介质（SQL/NoSQL/缓存/文件），彻底解耦。

## 👁️ 第四部分：AI 行为全量追踪与存储约束 (AI Traceability & Storage)

### 1. 契约定义原则（Contract Definition）
* **代码即契约**：系统内流动的所有数据流、接口、领域模型结构，必须强制在代码中直接使用 Zod 定义，通过代码编译自动获得类型与验证能力。

### 2. AI 操作追踪 (IOOR 协议)
* **全量流式记录**：每一次轮次必须作为原子单位进行全量记录；持久化允许在可配置的 ≤N 条 / ≤T ms 有界缓冲窗口内完成。受控 shutdown、执行完成、读前查询路径必须触发 flush；flush 失败必须进入 `audit_dead_letters`。非受控进程崩溃存在最多一个缓冲窗口的数据丢失风险，须在配置与修订日志中显式声明。（V1.5 修订，见文末修订日志）
* **IOOR 并发适配拓扑**：
  * **I (Input)**：大模型接收到的完整上下文快照。
  * **O (Output)**：大模型返回的原始响应，强控抓取独立的推理字段 (`reasoning_content`)。
  * **A (Action)**：AI 决策触发的工具调用，必须定义为**数组类型 (`tool_calls: [...]`)**以支持并发调用。
  * **R (Response)**：对应并发工具执行后返回的**结果数组 (`observations: [...]`)**。

### 3. 数据库存储与性能约束
* **混合存储（SQL + JSONB）**：状态、时间戳、追踪流水号作为独立标准 SQL 字段存储。高变动的业务 Payload、IOOR 详情存储在 `JSONB` 格式字段中（单个最大严格限制不超过 4MB，内部关键检索键建立 GIN 倒排索引）。
* **安全审计豁免**：基础业务持久化发生契约校验失败时抛出错误并拒绝。唯独 `domain/audit/`（审计域）拥有特权通道，允许降级将原始异常脏数据强行写入 JSONB 字段，死守错误現場。

### 4. 角色画像版本化约束
* **严禁覆盖更新（Append-Only）**：任何针对 Prompt、Agent 画像的调整必须作为新版本发布。系统加载时自动计算该文本内容的 SHA-256 动态哈希值（`profile_hash`），全量 Trace 审计日志必须强制绑定此 Hash。

### 5. 成本、性能审计与双重全链路脱敏
* **消费元数据**：必须记录 `model_provider`、`model_name`、`token_usage`（包含 `cached_prompt_tokens` 缓存命中数）以及网络耗时 `latency_ms`。
* **安全红线双脱敏管道**：
  1. **存储前置脱敏**：落库前过滤 Input/Output/Observation 中的密钥、密码及敏感用户隐私。
  2. **传输流式脱敏**：通过 **SSE 通道向客户端实时推送** 智能体思考流时，流量必须流经脱敏拦截层，严禁明文敏感信息外泄至浏览器端。

## 🚨 第五部分：异常处理与有界自愈闭环

* **拒绝静默失败（Fail-Fast）**：当大模型返回的数据彻底偏离契约时，阻断业务流并将任务状态更新为 `STUCK`。
* **有界自愈限制**：系统允许触发自动修复自愈（将报错字段和原因重新投喂给 LLM 要求其修正格式），但**自愈重试次数严格限制不超过 2 次**。2 次后若依然失败，系统彻底锁死并触发外部告警，强制唤醒人工干预。

## 📝 规范修订日志 (Revision Log)

### V1.5 (2026-05-20) — IOOR 持久化语义：实时直插 → 有界缓冲窗口

**修订条款**：第四部分 §4.2「全量流式记录」。

**原文**：「每一次轮次……必须作为一个原子单位进行**实时全量持久化存储**」。

**修订原因**：原条款的「实时直插」在高并发工作流下产生写放大（每 turn 一次 INSERT），是性能瓶颈。V1.5 引入 IOOR 批量缓冲（`ioorBuffer.js`）。

**诚实声明（bounded loss window）**：引入内存缓冲后，IOOR 不再是「写即落盘」。受控 shutdown、execution 完成、读前查询路径会强制 flush；flush 失败整批进 `audit_dead_letters`。**但非受控进程崩溃（`kill -9`、宿主机掉电、OOM Kill）下，最多丢失一个缓冲窗口（默认 ≤50 条 / ≤1000ms）的 IOOR 记录** —— 这是性能与「实时全量」之间的明确权衡，不是 bug。运维须知悉此风险窗口。

**配置**：`IOOR_BATCH_SIZE`（默认 50）/ `IOOR_BATCH_INTERVAL_MS`（默认 1000）。

**未变更**：「原子单位 / 全量记录 / 凡动必留痕」的本意不变——缓冲只改变「何时落盘」，不改变「是否落盘」与「落盘内容完整性」。
